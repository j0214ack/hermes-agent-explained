工具派發 · 連外部工具伺服器的那層

Tools 與 MCP：模型說「呼叫這個工具」之後，誰去執行、外部伺服器怎麼接進來

模型在一輪裡丟出一串工具呼叫，由 tool_executor 收下、派發給對應的處理函式執行，再把結果接回去。內建工具在本機跑；MCP 工具是接到一台外部工具伺服器、隔著協定來回。這頁把「派發邊界」跟「MCP 連線的生命週期與重連韌性」拆開看。誰丟出工具呼叫（對話迴圈本體）在「Agent 與學習」頁，這頁從派發那一刻接手。

誠實提醒

MCP client 的重連是寫死的有限重試。常數 _MAX_RECONNECT_RETRIES 直接定成 5（:279），伺服器任務的執行迴圈裡 if retries > _MAX_RECONNECT_RETRIES: 就 return、印一條 giving up 的警告，這台 MCP 伺服器就永久放棄、要到整個行程重啟才會再連（:2349）。沒有 config 開關可以調這個次數。這是真實的韌性上限：一台 MCP 伺服器若連續斷 5 次，這次行程剩下的時間它就不見了。

auth 失效跟 session 過期各有一條獨立的恢復重試路徑：_handle_auth_error_and_retry（:2539）會問 OAuth manager 能不能 refresh，能就重連並重試一次；_handle_session_expired_and_retry（:2708）token 還有效、只是伺服器端 session 過期，所以只觸發 transport 重連並重試一次，不走 OAuth。要留意：這兩條重試是對 5 個 MCP 操作（tools/call、resources/list、resources/read、prompts/list、prompts/get）一視同仁重試一次，並沒有針對 create／update／delete 這類有副作用的呼叫特別跳過。所以「一個改資料的工具呼叫被重試兩次」是可能發生的，原始碼這層沒有為了避免重複副作用而做防護，這點要誠實標出來。

連線不會自己保溫，要靠 keepalive。HTTP／SSE 的 session 會被伺服器端依它自己的 TTL 砍掉，所以 client 用 _DEFAULT_KEEPALIVE_INTERVAL = 180 秒（:291）的節奏送 ping 探活（_keepalive_probe，:1630）；伺服器若不支援 ping 就退而用 list_tools。idle 太久而沒及時 ping 的 session 一旦被砍，就要走前面那條重連。

tools/mcp_tool.py 約 4.7k 行，乍看很大，但這個大小是 MCP 協定本身的複雜度撐起來的、算合理：它要同時支撐 stdio 與 HTTP／SSE 三種傳輸、OAuth 認證與重連、sampling（伺服器反過來請 client 呼叫模型）、elicitation（伺服器請 client 跟使用者要結構化輸入）。每一塊都是協定明文要求的功能，不是雜訊堆積。
