Agent 與學習：跑完一輪對話，順手把記憶與技能也養起來

Gateway 把控制權交給 agent 之後，這頁接手。它把一輪對話的真正主體拆開：誰持有狀態、迴圈在哪、工具怎麼派發；再把「會學習」這件事拆成三個具體機制，記憶預取與回寫、自動建技能、技能策展，而不是當成單一魔法。總覽頁只用一張小卡帶過它，這頁放大看裡面的實體與關係。

這些方塊是 agent 子系統裡真實存在的「東西」。AIAgent 在左上當本體，對話迴圈是真正的迴圈主體（run_conversation 只是轉手），工具派發是一組模組函式（不是類別）。右側那群是「會學習」的三件事，記憶、技能、策展。每個方塊只標名字和一句目的，點方塊就從右側看完整說明（角色、關鍵方法、關係、限制）。把游標移到方塊上，只會點亮它牽到的關係線，其餘變淡。畫板比視窗大時，可在框內捲動。

同一批實體當生命線（lifeline），從左到右照一輪的流向排。箭頭是時間軸上的呼叫，由上往下發生。點上方的方塊、或點任一個帶編號的訊息步驟，右側開細節。虛線回程是結果或回覆往回送。這張是次要視角，主要看的是上面的實體圖。記憶回寫與策展刻意畫在最後，強調它們發生在回覆送出之後、不擋使用者。

誠實提醒

run_conversation 在 run_agent.py:5302 上看起來像是 agent 的主方法，其實只是一個轉手方法（forwarder）：它的本體就一句話，把所有參數原封不動轉給 agent.conversation_loop.run_conversation（:495）。真正的迴圈邏輯都在後者。在 run_agent.py 裡追迴圈會找錯地方，這是看這個 codebase 最容易踩的坑之一，整個檔到處是這種「Forwarder, see ...」的薄殼方法。

tool_executor 不是一個類別，是一組模組層級的函式。派發的入口是 agent 上的 _execute_tool_calls（run_agent.py:5188），它依這批工具看起來能不能平行，分流到 execute_tool_calls_concurrent（:260，唯讀工具走這條）或 execute_tool_calls_sequential（:792，單一呼叫或互動式工具走這條）。兩條最後都呼叫 model_tools.handle_function_call（model_tools.py:901）真正執行。完整的工具與 MCP 故事在 Tools 頁。

「會學習」不是一件魔法，是三件分開的事。一是記憶（memory）：每輪結束在背景把這輪回寫進記憶（sync）、並預取下一輪可能用到的記憶（prefetch）。二是自動建技能：agent 把成功的做法用 skill_manage 存成可重用的技能。三是策展（curator）：定期回顧這些技能。三者各有各的檔，混在一起講會以為是同一個東西。

Curator 只封存、不刪除。 curator.py 檔頭就寫明 only touches agent-created skills、never auto-deletes，only archives（:16）。它的範圍由 is_agent_created（tools/skill_usage.py:415）守住，內建與 hub 安裝的技能不在它管轄內；封存是把技能目錄搬進 .archive/，可復原，最大破壞動作就到這裡。
