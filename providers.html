Providers：agent 要一個模型，怎麼挑出後端、帶著後備把請求送出去

Hermes 不綁死單一模型供應商。agent 要跑一輪時只說「我要一個模型」，剩下的交給這層：先解析出該用哪一家供應商（原生支援 Anthropic 與約 28 家），從外掛目錄取出那家的 profile，必要時查線上模型目錄補資料，再帶著一池憑證（用完一把自動換下一把）把請求送到那家的後端。總覽頁只用一張小卡帶過它，這頁把它拆開看裡面的實體與關係。

誠實提醒

provider 解析不是查一張表就好。resolve_runtime_provider 是一個約 1.8k 行檔裡的真正 resolver：它要分辨 Azure Anthropic 捷徑、自訂供應商（custom:<name>）、明確指定的 base_url、本機伺服器（ollama / vllm…）、再到憑證池與環境變數的層層後備。圖上只畫主線：agent 要模型 → 解析挑出供應商 → 取 profile → 帶憑證送出。那些邊角分支都收在這個檔裡。

Hermes 原生支援 Anthropic 與約 28 家供應商：plugins/model-providers/ 底下目前正好 28 個外掛資料夾（anthropic、openai-codex、openrouter、gemini、deepseek、xai…），每個是一個自包含的 ProviderProfile 外掛。發現是惰性的：第一次呼叫 get_provider_profile 才掃描並匯入。使用者還能在 $HERMES_HOME/plugins/model-providers/ 放同名外掛覆蓋內建的（後寫者贏）。「約 28 家」指的是內建外掛數，不含線上模型目錄涵蓋的 100+ 家。

device-code OAuth 是 headless 部署的關鍵。沒有瀏覽器的環境（伺服器、容器）要拿 OpenAI Codex / Nous 這類 OAuth 授權，靠的是裝置碼流程：_request_device_code 先換一組 user_code 跟 verification_uri，使用者在別的裝置上開那個網址授權，程式這邊用 _poll_for_token 輪詢拿 token。openai-codex 的登入入口是 _login_openai_codex，token 落在 ~/.hermes/auth.json，之後進憑證池跟著輪替。
