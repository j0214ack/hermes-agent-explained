Gateway：把一則平台訊息變成一輪 agent 對話，再送回去

Gateway 是 Hermes 那個一直在跑的主控行程。它從某個聊天平台收到一則訊息，找出或建立這個 session 對應的 agent，備好歷史，驅動 agent 跑完一輪，再把回覆依各平台的送出規則送回去。總覽頁只用一張小卡帶過它，這頁把它拆開看裡面的實體與關係。

誠實提醒

gateway/run.py 是一個約 17.8k 行的巨型檔（god-file）。它這麼大不是因為核心邏輯複雜，多半是長年累積的多平台邊角情況：各平台的訊息格式、編輯能力、附件、執行緒、回覆路由限制都不一樣，這些差異一條一條疊在同一個檔裡。脊椎其實就是 _handle_message（:7265）這條路徑，圖上刻意只標主線錨點，把邊角情況留在外面。

APIServerAdapter 跑在 aiohttp 上，把 Hermes 對外變成一個 OpenAI 相容的 HTTP API。它的 API_SERVER_KEY 是單一一把 Bearer token：每個會執行 agent 的路由都先過 _check_auth（:950）用這把 key 比對，沒設這把 key 連伺服器都不會啟動。也就是說，這把 token 一旦外洩，等於把「能驅動 agent」的能力整個交出去，沒有更細的分權。

平台轉接器都是單租戶的 bot 轉接器：一個轉接器接一個 bot（一組 LINE channel token、一個 Telegram bot…），不是多租戶 SaaS 的概念。同一個 gateway 行程可以同時掛多個平台轉接器，但每個轉接器本身只服務它自己那一支 bot。
